Fairy Tale repo 読解メモ

強いAIの「魔法」ではなく、仕事の型を配る仕組み

fairy_tale repo は、Fable / Mythos 系の公開レポートから、普通の agent でも使える作業手順、検証ゲート、ベンチマーク改善ループを取り出す研究・配布 workspace です。

この記事の主張: Fairy Tale の本体はモデル能力ではなく、作業を「目的設定 → 証拠 → 小さな実行 → 検証 → 失敗分類 → 昇格」に分解する運用アルゴリズムであり、tako / SkillOps に持ち込むべき価値もそこにある。

発端: この repo は何をしているのか

たとえるなら、伝説の選手のバットを盗む場所ではなく、公開されている試合映像から「良い練習メニュー」を作る場所です。

Fable / Mythos という強い agent の公開レポートがある。Fairy Tale はそれを見て、「どんな道具を使ったのか」「失敗をどう直したのか」「どこで検証していたのか」を分解します。

ただし、秘密のモデル、重み、内部プロンプト、アクセス制限の突破は扱いません。公開情報から、再利用できる仕事の型だけを取り出します。

この図の読み方: Fairy Tale は「モデル再現」ではなく「仕事手順の抽出と配布」に責務を置いている。
やらない

秘密のAIを再現しない

アクセス制御、安全装置、輸出規制の迂回をしない。Fable / Mythos の中身ではなく、公開レポートから分かる作業プロセスだけを見る。

やる

作業の型を作る

予算を決める、証拠を残す、小さく試す、失敗を分類する、検証してから完了と言う。これを skill 化する。

守る

盛らない

ベンチマークの sample size、信頼区間、reference only、失敗分類を分けて書く。数字を強く見せるより、次の改善に使える形を優先する。

repo の構成: 作戦ノートと遠征バッグ

この repo は、研究ノート、配布パッケージ、検証器、公開ページが同居しています。最初は複雑に見えますが、役割で見ると分かりやすいです。

skills/ 正本の作戦ノート。Fairy Tale の思想、routing、禁止事項、検証ゲートが書かれている。
.agents/skills/ / .claude/skills/ Codex / Claude Code が読むための repo-local コピー。置き場所が違うのでコピーが必要になる。
plugins/fairy-tale/ 配布用の遠征バッグ。skills、scripts、schemas、hooks、resources、manifest をまとめる。
scripts/ 測定器と実験道具。benchmark runner、feedback ledger、fusion review、residency check がある。
adapters/ / schemas/ 外部ツールを安全に接続するための契約。入力、出力、証拠、安全境界、validation を決める。
sample_results/ 試合結果ノート。何を測り、どこで失敗し、どの feedback が効いたかを残す。

Fairy Tale の作業アルゴリズム

ここでいうアルゴリズムは、agent が迷子にならないための仕事の順番です。難しい数式ではなく、判断の流れです。

この画像の読み方: Fairy Tale の強みは、一発で正解することではなく、失敗を分類して次の手順へ戻す loop にある。
Fairy Tale の疑似コード

task を受け取る
  -> 何が成功か、何が危険かを決める
  -> 時間・tool・並列数の上限を決める
  -> README / test / schema / runner を小さく読む
  -> claim と source を分ける
  -> coding / legal / bio / security / UI など分野別 harness を選ぶ
  -> 小さく実行する
  -> test / runner / schema / screenshot で検証する
  -> 失敗を分類する
  -> 次回も使える rule だけを昇格する

ベンチマーク: 点数自慢ではなく、失敗を学習する測定器

ベンチマークは「試合の点数」です。ただし Fairy Tale は、点数だけではなく、条件、サンプル数、失敗の種類を重視します。

読み方の注意: README の多くの数値はローカル測定や reference 情報です。公式 leaderboard claim ではないものは、そのように明記されています。これは弱さではなく、あとで検証できるようにするための慎重さです。
分野 何を測るか Fairy Tale の見方 tako / SkillOps への示唆
Agentic coding SWE-Bench Pro 系。実 repo の bug を直せるか。 9/15 = 60.0%。ただし小 sample。失敗は API 互換、欠けた symbol、edge case、mock contract が中心。 成功率だけでなく「どの契約を壊したか」を failure taxonomy にする。
Biology BioMysteryBench preview。根拠ある生物系推論ができるか。 4/5 = 80.0%。根拠 gate により、勢いで断言する失敗を減らす方向。 専門領域は domain router と evidence table を先に作る。
Cybersecurity ExploitBench v8 sample。防御目的の sandbox 内でリスク探索できるか。 avg 1.33, 4/6 positive。Cap% の直接再現ではないため reference only。 security skill は weaponization ではなく、非攻撃的 evidence と再現可能な防御検証を出す。
Legal 法律タスクの criteria を満たせるか。 11/100 = 11.0%。低く見えるが、feedback retry で criterion pass や one-miss 減少を見る。 aggregate score より criterion-level feedback が skill 改善に効く。
Context 不完全な workspace で、無理に答えず安全に止まれるか。 guarded completion / abstention discipline を見る。 「答えない判断」も能力として評価する。

ベンチマーク改善ループ

1. baseline を測る
2. candidate skill ありで測る
3. 差を見る
4. 失敗を分類する
5. 失敗に効く小さな rule を作る
6. held-out sample で再測定する
7. 改善し、退行しない rule だけ keep
8. 悪化した rule は review / prune に戻す

ここで重要なのは、skill を「書いたから採用」しないことです。採用するには、測定と失敗分類と pruning が必要です。

plugin と SKILL.md の違い

SKILL.md は作戦ノート。plugin は、その作戦ノートと道具をまとめて配る遠征バッグです。

この画像の読み方: plugin の中心にも SKILL.md はある。ただし plugin は hook や script も一緒に届けられる。
観点 SKILL.md plugin
目的 agent の考え方と手順を変える skill と周辺道具をインストール可能にする
含むもの 文章、参照ファイル、軽い手順 skills、scripts、schemas、hooks、resources、manifest
失敗しやすい点 長文化、矛盾、測定なしの rule 追加 version drift、copy drift、hook 不発、manifest 不整合
SkillOps 的な位置づけ 評価・改善される content 本体 評価済み content を配る package

residency check: 古い作戦ノートで戦わないための検査

Fairy Tale には同じ skill のコピーが複数あります。これは危険なので、コピーが正本とズレていないかを検査します。

scripts/fairy_tale_residency_check.py は、skills/、plugins/fairy-tale/skills/、.agents/skills/、.claude/skills/ の一致を確認します。ズレは warning ではなく failure です。

residency check の考え方

canonical = skills/
  -> plugin copy と一致するか
  -> Codex repo-local copy と一致するか
  -> Claude Code repo-local copy と一致するか
  -> manifest が ./skills/ を指すか
  -> benchmark runner が Fairy Tale 前提を忘れていないか

全部 OK なら、長い benchmark / multi-agent 作業に入ってよい

tako / SkillOps にどう活かすか

一番大きな学びは、「skill を書く」ではなく「skill を測って昇格させる」です。

SkillOps

PromotionGate を主役にする

ms feedback や感想だけで採用しない。fixture、hidden eval、deterministic verifier、owner review を通したものだけ昇格する。

tako

runtime と content を分ける

tako は skill を実行・配布・観測する runtime。SKILL.md は content。runtime 固有事情を skill 本文に混ぜすぎない。

plugin

配布単位として扱う

plugin は便利なフォルダではなく、version、manifest、hooks、scripts、schemas を持つ release artifact。CI で壊れないことを確認する。

運用

residency を preflight に入れる

長い agent 作業では、正しい skill が resident であること自体を検証対象にする。これを tako 側にも持ち込みたい。

SkillOps に写すなら、この順番

1. candidate skill を作る
2. どの failure class を直す skill かを書く
3. fixture / eval pack / verifier を作る
4. baseline を測る
5. candidate ありで測る
6. 改善したか、退行していないかを見る
7. hidden test と owner review を通す
8. registry に promotion metadata を残す
9. plugin / installer / ms index で配布確認する
10. 別 Mac でも install -> load -> run を確認する

今の構成はベストか

研究・配布・検証を一つの repo で高速に回す段階なら、今の構成はかなり合っています。ただし最終形としては CI が足りません。

1良い: skills/ 正本と複数コピーを residency check で守っている。
2良い: ベンチマークを盛らず、reference only や CI 幅を分けている。
3良い: adapter schema によって外部ツールを雑に混ぜない。
4弱い: コピーが多く、保守コストがある。check がなければ危険。
5次: GitHub Actions で residency、adapter validate、package build、installer dry-run を回したい。
結論: いまは repo 分割より CI 追加が先。利用者が増えて release 管理が重くなったら、研究 repo と配布 repo の分離を検討する。
